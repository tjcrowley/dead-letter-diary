```sql
CREATE TABLE notification_thresholds (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

```

<!-- FIXED INSERT should be: -->
```typescript
// CORRECT: columns threshold_minutes and tone
const thresholds = [
  { minutes: 24 * 60,      tone: "gentle" },  // 1440 minutes
  { minutes: 4 * 60,       tone: "urgent" },  //  240 minutes
  { minutes: 1 * 60,       tone: "urgent" },  //   60 minutes
  { minutes: 15,           tone: "final"  },  //   15 minutes
];

await fastify.pg.query(
  `INSERT INTO notification_thresholds (user_id, threshold_minutes, tone)
   VALUES ($1, $2, $3) ON CONFLICT DO NOTHING`,
  [userId, t.minutes, t.tone]
);
```
Note: ON CONFLICT DO NOTHING has no named constraint — it silently skips on any conflict.
This is acceptable since threshold_minutes does NOT have a unique constraint in the schema,
so ON CONFLICT DO NOTHING here will never actually conflict. It is defensive only.
The real fix is the column names.

<!-- notifications table schema (lines 89-95) -->
```sql
CREATE TABLE notifications (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id      UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  subscription JSONB NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);
-- MISSING: unique index on (user_id, (subscription->>'endpoint'))
-- Without this, the ON CONFLICT in notifications.ts throws at runtime
```

<!-- Required index to add -->
```sql
CREATE UNIQUE INDEX idx_notifications_user_endpoint
  ON notifications(user_id, (subscription->>'endpoint'));
```
This replaces the existing non-unique idx_notifications_user index (line 113 in migration).
Both can coexist — keep the existing idx_notifications_user index as well.

<!-- ON CONFLICT clause in notifications.ts lines 35-41 -->
```typescript
// This is structurally valid once the unique index exists.
// PostgreSQL requires a unique constraint/index matching the ON CONFLICT columns exactly.
ON CONFLICT ((user_id), (subscription->>'endpoint'))
// With the new UNIQUE INDEX, PostgreSQL will resolve this correctly.
```
Note: The ON CONFLICT syntax `((user_id), (subscription->>'endpoint'))` uses redundant parens
around user_id. PostgreSQL accepts this. No change needed to notifications.ts itself.
