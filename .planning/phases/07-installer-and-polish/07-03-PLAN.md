<objective>
Implement read-only past entries browser with client-side decryption (WRITE-06). The server has the ciphertext but not the DMK — decryption always happens in the browser using the session's in-memory key.

Purpose: Users need to be able to read what they wrote. Without this, the diary is write-only, which defeats the point. The E2E encryption model means all decryption is client-side.

Output: GET /api/entries (list) and GET /api/entries/:id (single) added to entries.ts; new /entries page with list + decrypt-on-click.

  <action>
    Add two new routes to the existing `entries.ts` file (after the existing POST /api/entries):

    **`GET /api/entries`** — list route, requires auth.
    Query: `SELECT id, word_count, created_at FROM entries WHERE user_id = $1 ORDER BY created_at DESC LIMIT 100`
    Return: `{ entries: Array<{ id: string, word_count: number, created_at: string }> }`
    Do NOT return ciphertext in the list — that would be unnecessary bandwidth and is unnecessary for listing.

    **`GET /api/entries/:id`** — single entry route, requires auth.
    ```typescript
    interface EntryParams { id: string }
    ```
    Query: `SELECT id, ciphertext, iv, aad, word_count, created_at FROM entries WHERE id = $1 AND user_id = $2`
    The `AND user_id = $2` is critical — prevents one user reading another's ciphertext.
    If not found, return 404.
    Return the ciphertext/iv/aad as base64url strings:
    ```typescript
    return {
      id: row.id,
      ciphertext: Buffer.from(row.ciphertext).toString('base64url'),
      iv: Buffer.from(row.iv).toString('base64url'),
      aad: Buffer.from(row.aad).toString('base64url'),
      word_count: row.word_count,
      created_at: row.created_at,
    }
    ```

    Both routes use `preHandler: [requireAuth]`.

    Validation Architecture: `GET /api/entries` with valid session returns `{ entries: [] }` for new user. `GET /api/entries/nonexistent-id` returns 404. Verified by curl with session cookie.

  <name>Task 2: Past entries browser page with client-side decryption</name>
  <files>apps/web/app/entries/page.tsx</files>
  <action>
    Create `apps/web/app/entries/page.tsx` — a `"use client"` component.

    **State:**
    - `entries: Array<{ id: string, word_count: number, created_at: string }>` — list from API
    - `selectedId: string | null` — which entry is expanded
    - `decryptedText: string | null` — plaintext after decryption
    - `decrypting: boolean` — spinner state
    - `decryptError: string | null` — if DMK missing or decryption fails
    - `loading: boolean` — initial list load

    **On mount:** `fetch('/api/entries', { credentials: 'include' })` → populate entries. If 401, show "Please unlock your diary first" with a link to /unlock. If empty list, show "No entries yet — start writing!" with link to /write.

    **Entry list rendering:**
    Each entry renders as a clickable row showing: formatted date (`new Date(created_at).toLocaleDateString(undefined, { weekday: 'long', year: 'numeric', month: 'long', day: 'numeric' })`) and word count (`{word_count} words`). Clicking a row triggers decryption.

    **On entry click (`handleDecrypt(id)`):**
    1. Check `getDMK()` — if null, set decryptError: "Your diary is locked. Unlock it first to read entries." Return early.
    2. Set decrypting=true, selectedId=id, decryptedText=null, decryptError=null.
    3. `fetch('/api/entries/' + id, { credentials: 'include' })` — get { ciphertext, iv, aad }.
    4. Decode base64url bytes: `Uint8Array.from(atob(b64url.replace(/-/g,'+').replace(/_/,'/')), c => c.charCodeAt(0))` — or use a helper.
    5. `crypto.subtle.decrypt({ name: 'AES-GCM', iv: ivBytes, additionalData: aadBytes }, dmk, ciphertextBytes)`.
    6. `new TextDecoder().decode(plainBuf)` → setDecryptedText.
    7. Wrap in try/catch — on error set decryptError: "Decryption failed. This entry may be from a different session key."
    8. Set decrypting=false.

    **Decrypted view:** When decryptedText is set, render it in a `<pre>` block below the selected entry row (inline expansion, not a modal). Style: `{ whiteSpace: 'pre-wrap', fontFamily: 'inherit', backgroundColor: '#111', padding: '1rem', borderRadius: '6px', color: '#e8e8e8', marginTop: '0.5rem' }`. Include a "Close" button that clears decryptedText and selectedId.

    **No edit affordance:** Strictly read-only. No textarea, no save button, no edit link.

    **Wipe guard:** Import `getDMK` from `@/lib/session-dmk`. If entries fetch returns entries but DMK is null, show a banner: "Unlock your diary to decrypt entries" — the list still shows (metadata only), but clicking any entry will hit the DMK null guard.

    **Page heading:** Display the diary name if available (fetch GET /api/settings on mount, use diary_name field). Fallback: "Past Entries". Sub-heading: "Read-only — entries are decrypted in your browser".

    Validation Architecture: Page renders at /entries. List loads without plaintext. Click triggers fetch + decrypt. DMK-null case shows error without crash. Verified by checkpoint:human-verify in phase verification.
