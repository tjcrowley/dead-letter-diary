**apps/api/src/boot/secrets.ts:** Follow RESEARCH.md Pattern 3, adapted for Docker environment.

Export `async function ensureSecrets()` that:
1. Checks `process.env.SESSION_SECRET` -- if missing, generates `crypto.randomBytes(32).toString('hex')` and sets it on `process.env`
2. Checks `process.env.VAPID_PUBLIC_KEY` -- if missing, calls `webpush.generateVAPIDKeys()` and sets both VAPID_PUBLIC_KEY and VAPID_PRIVATE_KEY on `process.env`
3. Checks `process.env.SHARD_ENCRYPTION_KEY` -- if missing, generates `crypto.randomBytes(32).toString('hex')` and sets it on `process.env`
4. Logs which secrets were generated (e.g., "Generated SESSION_SECRET", "Generated VAPID keys")
5. If ANY secrets were generated, write them to a `.env.generated` file in the project root (or `/app/.env.generated` in Docker) so they persist across container restarts. Use a Docker volume or bind mount for persistence.

IMPORTANT: Do NOT write to the host's `.env` file from inside Docker (bind mount permission issues, race condition pitfall from RESEARCH.md). Instead:
- Generate secrets into `process.env` for the current run
- Write a `.env.generated` file to a persistent volume so secrets survive restarts
- On subsequent boots, load `.env.generated` before checking for missing secrets
- Alternative simpler approach: generate secrets and log them to stdout with a message like "Add these to your .env file", and also set them in process.env for the current session. The key requirement is that secrets exist in memory during runtime.

Choose the approach that works most reliably in Docker. The simplest correct approach: check env vars, generate missing ones into process.env, log that they were generated. For persistence, write to a file on a Docker volume.

**HTTPS boot check (INST-10):** Add to server.ts startup sequence, AFTER secrets and BEFORE listen:
- Check if `SITE_HOSTNAME` is set and not `localhost`
  - If hostname is not localhost, the app MUST be accessed via HTTPS. Log a warning: "HTTPS required for non-localhost deployments. Caddy provides HTTPS automatically."
- Add a Fastify `onRequest` hook that checks the `x-forwarded-proto` header (set by Caddy):
  - If proto is `http` AND hostname is NOT `localhost` AND NOT `127.0.0.1`, return 403 with `{ error: "HTTPS required" }`
  - Allow `http` for `localhost` / `127.0.0.1` (dev exception per INST-10)
- This check works because Caddy sets `X-Forwarded-Proto` when proxying

**Update server.ts** startup sequence:
1. Load dotenv
2. Call `ensureSecrets()` (generate/load secrets)
3. Run migrations
4. Register db plugin
5. Register HTTPS check hook
6. Register health routes
7. Listen
